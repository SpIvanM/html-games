# Toddler Keyboard Fireworks Game

Клавиатурные фейерверки для малышей

## Что это

Это полноэкранная браузерная игра для маленького ребенка, в которой любое нажатие клавиши превращается в крупный визуальный экшен на экране. Игра рассчитана на простой сценарий: ребенок нажимает любые клавиши подряд, а браузер показывает большие яркие символы, вспышки, частицы, движение и постепенное затухание.

Проект сделан как статическое приложение без runtime-зависимостей: одна HTML-страница, CSS для оформления и несколько ES-модулей на JavaScript.

## Детальное ТЗ

### 1. Цель продукта

Сделать очень простую интерактивную игру для ребенка возраста около 1 года, где основное действие — любое нажатие на клавиатуре вызывает красивый визуальный отклик на весь экран.

### 2. Основной пользовательский сценарий

1. Взрослый открывает страницу игры в браузере.
2. Игра занимает всю зону браузера.
3. После первого взаимодействия игра пытается перейти в полноэкранный режим.
4. Ребенок нажимает любые клавиши на клавиатуре.
5. Каждая клавиша сразу порождает на экране собственный визуальный объект.
6. Объект показывает крупную букву или понятную подпись клавиши, запускает вспышку и частицы, выполняет собственное движение и потом полностью исчезает.
7. Если нажато несколько клавиш подряд или почти одновременно, эффекты должны существовать параллельно, не мешая друг другу и не заменяя друг друга.

### 3. Требования к игровой зоне

- Игра должна работать на всю доступную зону браузера.
- Интерфейс должен быть максимально простым: без меню, счета, уровней и сложных текстов.
- На старте допускается только простой экран-приглашение для взрослого пользователя.
- Визуальный стиль должен быть ярким, дружелюбным и заметным для маленького ребенка.

### 4. Требования к клавиатурному вводу

- Нужно принимать все возможные клавиши, которые браузер реально отдает в JavaScript.
- Любая принятая клавиша должна вызывать визуальное действие.
- Должны учитываться не только буквы и цифры, но и служебные клавиши: `Space`, `Enter`, `Shift`, `Control`, `Alt`, стрелки, `Meta` и другие доступные клавиши.
- Клавиша `F11` не должна перехватываться, потому что она нужна для выхода из полноэкранного режима браузера.
- Клавиша `F12` и системные клавиши ОС желательно перехватывать, но это допускается только в пределах возможностей браузера.
- Игра должна делать best effort по удержанию фокуса и подавлению стандартного поведения браузера, чтобы случайные нажатия как можно реже выводили пользователя из игрового режима.

### 5. Требования к представлению клавиш

- На экране должна появляться либо сама буква/цифра, либо читаемая подпись клавиши.
- Для служебных клавиш подписи должны быть дружественными и понятными, например:
  - `Space` -> `SPACE`
  - `Enter` -> `ENTER`
  - `Shift` -> `SHIFT`
  - `Control` -> `CTRL`
  - `Meta` -> `WIN`
  - `ArrowLeft` -> `LEFT`
- Одна и та же клавиша должна всегда отображаться одинаково.

### 6. Требования к эффектам

- Каждый эффект должен появляться мгновенно после нажатия клавиши.
- Эффект должен включать:
  - крупный цветной символ или подпись;
  - вспышку или сияние;
  - частицы или блестки;
  - заметное движение;
  - плавное затухание;
  - полное удаление со сцены после завершения жизни эффекта.
- Разные клавиши могут иметь разные типы поведения.
- Поведение клавиши должно быть постоянным. Если конкретная клавиша однажды сопоставлена с определенным типом эффекта, то она всегда использует именно его.

### 7. Примеры допустимых типов поведения

ТЗ требует разнообразия поведения букв и клавиш. Реализация должна поддерживать идею постоянного характера клавиши. Возможные примеры поведения:

- плавное всплытие вверх;
- подпрыгивание, как зайчик;
- вращение по орбите;
- отскакивание от стенок;
- пульсация с фейерверком.

### 8. Требования к параллельности

- Если ребенок нажимает много клавиш подряд, каждая клавиша должна создавать собственный независимый объект.
- Эффекты должны выполняться одновременно.
- Новое нажатие не должно уничтожать уже существующие эффекты.
- После завершения жизни каждый эффект должен исчезать самостоятельно.

### 9. Ограничения платформы

- Решение должно работать в обычном браузере.
- Нельзя гарантированно заблокировать системные клавиши Windows, отдельные DevTools shortcuts и некоторые комбинации уровня ОС.
- Игра должна честно работать в режиме best effort: перехватывать все, что доступно в браузере, и не ломать обязательный выход через `F11`.

## Что реализовано

### Формат приложения

Реализация сделана как статическая браузерная игра:

- `index.html` — корневая страница;
- `src/styles.css` — оформление сцены, overlay и эффектов;
- `src/main.js` — точка входа;
- `src/game.js` — игровой контроллер и цикл;
- `src/keyboard.js` — нормализация клавиш и правила перехвата;
- `src/effects.js` — движок эффектов и анимаций.

Runtime-зависимостей нет. Dev-зависимости используются только для тестов и локального запуска.

### Поведение на старте

При открытии страницы показывается стартовый overlay с короткой подсказкой. После первого клика или допустимого нажатия клавиши игра:

1. помечается как armed;
2. скрывает overlay;
3. пытается запросить Fullscreen API;
4. переводит фокус на игровую сцену.

Если окно теряет фокус, игра показывает overlay повторно с подсказкой о возврате в игру и снова пытается удерживать фокус, когда браузер позволяет это сделать.

## Архитектура

### 1. Слой разметки

`index.html` определяет минимальный DOM-каркас:

- `game-root` — общий контейнер игры;
- `stage` — игровая зона на весь экран;
- `effects-layer` — слой, в который динамически добавляются эффекты;
- `launch-overlay` — стартовая и refocus-подсказка.

### 2. Слой ввода

`src/keyboard.js` отвечает за три вещи:

- нормализация `KeyboardEvent.key` в стабильное внутреннее имя клавиши;
- форматирование дружественной подписи для рендера;
- решение, нужно ли подавлять стандартное поведение браузера.

Ключевые правила:

- одиночные печатные символы приводятся к нижнему регистру для стабильности;
- известные алиасы браузеров приводятся к одному имени, например `Spacebar` -> `Space`, `OS` -> `Meta`, `Esc` -> `Escape`;
- `F11` пропускается браузеру;
- остальные распознанные клавиши перехватываются через `preventDefault()`, если событие `cancelable`.

### 3. Слой игровой логики

`src/game.js` содержит класс `ToddlerFireworksGame`, который:

- подписывается на `keydown`, `resize`, `blur`, `focus`, `visibilitychange`;
- запускает игру после первого взаимодействия;
- создает эффект на каждое допустимое нажатие клавиши;
- ведет коллекцию активных эффектов;
- обновляет и удаляет эффекты в цикле `requestAnimationFrame`.

Логика специально сделана через небольшие отдельные модули, чтобы ее можно было тестировать без запуска реального браузера вручную.

### 4. Слой эффектов

`src/effects.js` отвечает за все визуальное поведение:

- сопоставляет клавишу с типом эффекта и палитрой;
- создает состояние эффекта;
- обновляет его во времени;
- рендерит состояние в DOM-элементы.

Архитектурно эффект состоит из:

- базовой информации о клавише;
- текущего положения и параметров трансформации;
- палитры цвета;
- массива частиц;
- lifetime и флага завершения.

## Как работает детерминированное сопоставление клавиш

Игра не хранит таблицу "каждая клавиша вручную -> отдельный эффект". Вместо этого используется детерминированный hash по нормализованному имени клавиши.

Это дает следующие свойства:

- одна и та же клавиша всегда получает один и тот же `effectType`;
- одна и та же клавиша всегда получает одну и ту же цветовую палитру;
- разные клавиши естественным образом распределяются по нескольким типам поведения;
- система легко расширяется новыми эффектами без ручной настройки каждой клавиши.

Поддерживаемые типы поведения в текущей реализации:

- `float-pop` — плавно всплывает и покачивается;
- `rabbit-hop` — подпрыгивает и двигается как "зайчик";
- `spin-orbit` — вращается по орбите;
- `wall-bounce` — летает и отскакивает от границ сцены;
- `pulse-burst` — пульсирует и уходит с фейерверком.

## Жизненный цикл одного эффекта

1. Пользователь нажимает клавишу.
2. `game.js` получает `keydown` и нормализует клавишу.
3. Если клавиша не `F11`, событие по возможности подавляется.
4. `spawnEffect()` создает объект эффекта:
   - выбирает позицию в безопасной зоне сцены;
   - назначает тип движения;
   - назначает палитру;
   - создает частицы;
   - задает время жизни.
5. Для эффекта создается DOM-структура в `effects-layer`.
6. На каждом кадре `updateEffect()` изменяет позицию, угол, масштаб, прозрачность и состояние частиц.
7. `renderEffect()` переносит вычисленное состояние в CSS-style DOM-элементов.
8. Когда эффект и его частицы полностью завершены, он удаляется из DOM и из внутренней коллекции.

## Визуальная реализация

Визуальная система построена на DOM + CSS + JavaScript-анимации, а не на canvas.

Каждый эффект включает:

- корневой элемент `effect`;
- свечение `effect__aura`;
- кольцо-вспышку `effect__burst`;
- крупную подпись `effect__label`;
- набор частиц `effect__particle`.

Преимущества такого подхода:

- проще отлаживать DOM в браузере;
- можно тестировать наличие эффектов через Playwright и jsdom;
- удобно использовать CSS-переменные для палитры и свечения;
- легче поддерживать понятную структуру без canvas-рендеринга.

## Логика обработки клавиш

### Нормализация

Внутренний идентификатор клавиши строится по `event.key`, а не по физическому `event.code`. Это значит:

- поведение зависит от того, какая клавиша фактически пришла в браузер;
- один и тот же символ имеет стабильное поведение;
- раскладка влияет на результат логично и прозрачно для пользователя.

### Подавление стандартного поведения

Игра делает best effort, чтобы клавиши оставались внутри игрового опыта:

- для распознанных клавиш вызывается `preventDefault()`, если это разрешено браузером;
- `F11` намеренно не блокируется;
- `F12`, системные клавиши Windows и часть комбинаций ОС могут остаться вне контроля приложения из-за ограничений браузерной среды.

## Поток работы приложения

1. Страница загружается и подключает `src/main.js`.
2. `main.js` проверяет, что все нужные DOM-узлы существуют.
3. Создается экземпляр `ToddlerFireworksGame`.
4. Контроллер подписывается на браузерные события.
5. После первого взаимодействия игра считается активированной.
6. Каждое нажатие клавиши порождает новый независимый эффект.
7. Глобальный animation loop обновляет все эффекты параллельно.
8. Завершенные эффекты удаляются, чтобы не было бесконечного накопления DOM-узлов.

## Детали реализации по модулям

### `src/main.js`

Отвечает только за bootstrap:

- находит DOM-элементы;
- проверяет, что разметка полная;
- создает контроллер игры;
- запускает `attach()`.

### `src/game.js`

Отвечает за orchestration:

- жизненный цикл игры;
- fullscreen attempt;
- фокус;
- overlay;
- коллекцию активных эффектов;
- requestAnimationFrame loop.

### `src/keyboard.js`

Отвечает за контракты ввода:

- `normalizeKey(event): string | null`
- `formatKeyLabel(key): string`
- `shouldAllowBrowserDefault(event): boolean`
- `shouldPreventDefault(event): boolean`

### `src/effects.js`

Отвечает за контракты эффекта:

- `resolveEffectForKey(key)`
- `spawnEffect(key, timestamp, bounds, randomSource)`
- `updateEffect(instance, dt, bounds)`
- `renderEffect(instance, elements)`

## Проверяемые требования и соответствие реализации

- Полноэкранная игровая зона: реализовано через `game-shell`, `stage` и попытку `requestFullscreen()`.
- Прием почти всех клавиш: реализовано через глобальный `keydown` и нормализацию `event.key`.
- Исключение для `F11`: реализовано явно.
- Постоянный характер клавиши: реализовано через hash-based deterministic mapping.
- Параллельные эффекты: реализовано через `Map` активных эффектов.
- Полное исчезновение после жизни эффекта: реализовано через `expired` и удаление DOM-узла.
- Разные визуальные типы поведения: реализовано через 5 effect families.
- Подавление стандартного поведения браузера: реализовано в рамках возможностей браузера.

## Ограничения и честные допущения

- Браузер не может гарантированно перехватить все системные клавиши и все сочетания уровня ОС.
- Windows key, некоторые DevTools shortcuts и часть служебных клавиш могут обрабатываться системой раньше JavaScript.
- Fullscreen API может быть отклонен браузером, автоматизацией или пользовательскими настройками.
- В таком случае игра продолжает работать в максимальной площади окна, но без гарантированного настоящего fullscreen.

## Запуск проекта

### Локальный запуск

```bash
npm install
npm run dev
```

После этого игру можно открыть по адресу `http://127.0.0.1:3000`.

### Тесты

```bash
npm test
```

Команда запускает:

- unit tests и coverage через Vitest;
- browser E2E tests через Playwright.

## Подход к тестированию

Реализация проверяется тремя уровнями тестов:

### 1. Unit tests

Проверяют:

- нормализацию клавиш;
- форматирование подписей;
- правило исключения для `F11`;
- стабильность сопоставления клавиши и эффекта;
- жизненный цикл эффектов;
- обновление частиц;
- DOM-style rendering в `renderEffect()`.

### 2. Browser-like unit tests через jsdom

Проверяют:

- bootstrap приложения;
- старт по клику;
- появление эффектов от клавиш;
- скрытие и показ overlay;
- удаление завершенных эффектов;
- поведение при отсутствии DOM-узлов.

### 3. End-to-end tests через Playwright

Проверяют реальный браузерный сценарий:

- первый запуск и скрытие overlay;
- появление эффекта от обычной клавиши;
- одновременное существование нескольких эффектов;
- подавление стандартного поведения для не-`F11`;
- полное исчезновение эффекта после fade-out.

## Возможные дальнейшие улучшения

- добавить звук на каждую клавишу;
- добавить отдельные палитры для разных групп клавиш;
- добавить canvas-режим для еще более тяжелых частиц;
- добавить kiosk-оболочку вне обычного браузера, если критично жестче блокировать системные клавиши;
- добавить переключаемые режимы для разных возрастов.

## Краткий итог

Проект реализует заявленное ТЗ как браузерную полноэкранную игру, где каждое нажатие клавиши превращается в яркий независимый визуальный эффект. Архитектура разделена на модуль ввода, модуль игровых состояний и модуль эффектов, а ограничения системных клавиш обработаны честно: все, что браузер разрешает перехватить, перехватывается, при этом `F11` специально оставлена для безопасного выхода из полноэкранного режима.
